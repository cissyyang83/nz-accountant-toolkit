# Maintenance Plan

This document records how the repository will show evidence of active maintenance.

## Regular Maintenance Activities

| Activity | Frequency | Evidence |
| --- | --- | --- |
| Review new issues | Fortnightly where possible | Issue comments, labels, closed issues |
| Review pull requests | Fortnightly where possible | PR reviews and merge history |
| Update roadmap | Quarterly | Roadmap updates |
| Publish releases | When stable improvements are ready | GitHub releases and changelog |
| Review tax-sensitive wording | Before release | Release checklist and reviewer notes |
| Run tests | On every pull request | GitHub Actions |

## Release Checklist

Before publishing a release:

- Update `CHANGELOG.md`.
- Update `docs/release-v0.1.md` or create a new release note.
- Run tests.
- Check that examples contain no client data.
- Check tax-sensitive content for appropriate disclaimers.
- Confirm known limitations are documented.

## Maintenance Evidence Still Needed

The repository should build real evidence over time:

- Public GitHub repository URL.
- Stars or forks.
- Issues opened and triaged.
- Pull requests reviewed.
- Releases tagged.
- Users or firms giving feedback.
- Download or clone activity if available.
